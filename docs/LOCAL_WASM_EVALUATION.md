# Local WebAssembly evaluation

How arcjet-go evaluates local rules (bot detection, email validation,
filter matching, sensitive info detection) via the shared
`arcjet_analyze_js_req` WebAssembly component — the same module
[`@arcjet/analyze-wasm`][js-analyze] and [`arcjet._analyze`][py-analyze]
load. All three SDKs and the Decide service make identical local
decisions because they share these bytes.

[js-analyze]: https://github.com/arcjet/arcjet-js/tree/main/analyze-wasm
[py-analyze]: https://github.com/arcjet/arcjet-py/tree/main/src/arcjet/_analyze

## What lives where

```text
internal/local/jsreq/
├── README.md          # regen instructions (developer-facing)
├── bindings.go        # GENERATED by gravity; do not edit
├── imports.go         # hand-written: NewFactory + host callback defaults
├── imports_test.go    # in-package unit tests for the import layer
└── js_req.wasm        # wizered core wasm, embedded via //go:embed

local_decision.go      # localEvaluator: factory lifecycle + per-rule eval
```

The package name `jsreq` is hand-edited after gravity emits `js_req` —
see `internal/local/jsreq/README.md`.

## Public seam: `jsreq.NewFactory(ctx, Callbacks)`

The package exposes one constructor:

```go
factory, err := jsreq.NewFactory(ctx, jsreq.Callbacks{
    DetectSensitiveInfo: customDetect, // optional
})
```

This mirrors arcjet-py's `wire_imports(linker, component, callbacks)`
and arcjet-js's `createCoreImports(detect)`. Callbacks in, ready
factory out. The five generated host-import interfaces
(`IJsReqEmailValidatorOverrides`, `IJsReqSensitiveInformationIdentifier`,
`IJsReqVerifyBot`, `IJsReqBotIdentifier`, `IJsReqFilterOverrides`) are
package-private; the call site never threads them positionally.

Every callback defaults to fail-open behaviour:

| Import callback                      | Default            |
| ------------------------------------ | ------------------ |
| `is-free-email` / `is-disposable`    | `Unknown`          |
| `has-mx-records` / `has-gravatar`    | `Unknown`          |
| `verify-bot.verify`                  | `Unverifiable`     |
| `bot-identifier.detect`              | `nil` (no extras)  |
| `filter-overrides.ip-lookup`         | `nil`              |
| `sensitive-information-identifier.detect` | one `nil` per token |

The wasm analyzer treats `Unknown` / `nil` as "skip", so unset callbacks
never tip a decision into deny.

### Custom sensitive-info detect

Public API:

```go
type SensitiveInfoDetect func(ctx context.Context, tokens []string) []EntityType
```

Set on `Config.SensitiveInfoDetect` or `GuardConfig.SensitiveInfoDetect`.
Empty `EntityType` slots mean "unclassified"; any other value (built-in
constant or custom label) is recorded. The SDK adapter translates the
public `[]EntityType` to the wasm-side `[]SensitiveInfoEntity`, picking
the right built-in variant (`Email`, `PhoneNumber`, `IpAddress`,
`CreditCardNumber`) or `SensitiveInfoEntityCustom{Value: ...}` for
custom labels.

The wasm config's `SkipCustomDetect` flag flips automatically: stays
`true` when no callback is configured (saves wasm cycles), goes `false`
when a callback is set.

## Lifecycle: `localEvaluator`

```go
type localEvaluator struct {
    mu        sync.Mutex
    factory   *jsreq.JsReqFactory
    closed    bool
    callbacks jsreq.Callbacks
}
```

* `newLocalEvaluator` — used by `Client`. Eagerly compiles the factory
  if any rule reports a `localKind`. Compilation is the expensive step
  (~hundreds of ms); doing it at construction keeps the first request
  fast.
* `newLazyLocalEvaluator` — used by `GuardClient`. Defers compilation
  until the first call that needs it, because `GuardClient` doesn't see
  its rules until each `Guard` call.
* `factoryLazy` — single entry point for both modes. Holds `mu` only
  long enough to compile-or-return, never across instantiation.
* `instance` — `factory.Instantiate(ctx)` runs outside the lock so
  concurrent requests don't serialise on it. Returns a `release`
  closure for `defer`.
* `close` — sets `closed = true` and drops the factory. Subsequent
  calls return `ErrWasmClosed`.

### One instance per call

Wazero (and the JS / Python wasm runtimes the reference SDKs use)
require fresh instances per invocation — after a wasm function call the
instance's state is spent. The factory caches the *compiled* module;
the per-call work is just instantiation, which is cheap (~ms). This
matches arcjet-py's `Component.from_file` (compile once) + fresh
`Store`/`Instance` per call.

## Rule wiring

### HTTP-path rules (in `local_decision.go`)

Each `Rule` from `rules.go` reports a `localKind` bit; `validateEmail`
/ `detectBot` / `matchFilter` / `detectSensitiveInfo` on
`localEvaluator` call into the corresponding `inst.IsValidEmail` /
`inst.DetectBot` / `inst.MatchFilters` / `inst.DetectSensitiveInfo`
generated method. Returns from the wasm are records (not JSON strings)
so no `json.Unmarshal` is needed on the way out.

### Guard-path rules (in `guard.go`)

The `GuardSensitiveInfoRule.Text(text)` callback runs the analyzer
locally via the same factory, then submits:

```json
{
  "localSensitiveInfo": {
    "configEntitiesAllow": {"entities": ["..."]}  // OR configEntitiesDeny
    "inputTextHash": "<sha256-hex>",
    "resultComputed": {
      "conclusion": "GUARD_CONCLUSION_ALLOW|DENY",
      "detected": true,
      "detectedEntityTypes": ["EMAIL", ...]
    },
    "resultDurationMs": 12
  }
}
```

The raw text never reaches the server. Wire shape mirrors
arcjet-py's `guard/_convert.py:RuleLocalSensitiveInfo`.

The unexported `guardSubmission(ctx, *localEvaluator)` method is the
seam that lets Guard rules optionally use the shared factory; all
other rule kinds ignore the evaluator.

## Build pipeline

The `internal/local/jsreq/js_req.wasm` artefact comes from
`arcjet/arcjet-analyze/bindings_js_req/dist/arcjet_analyze_js_req.wasm`
(post-wizer, pre-`wasm-tools component new`). The component-form wasm
that ships in arcjet-js / arcjet-py is wrong for arcjet-go because
wazero only speaks core wasm.

### Why wizer matters

`bindings_js_req` exports a `wizer.initialize` function that populates
the `USER_AGENT_PARSER` static. Without that pre-init, bot detection
traps with `"failed to detect bot: user agent parser unavailable"`.
The arcjet-analyze Makefile runs:

```
cargo build --release → core wasm (USER_AGENT_PARSER = None)
↓ wizer --init-func wizer-initialize
↓ wasm-opt -O3                  (optional size optimisation)
↓ wasm-tools component new      (only needed for the JS/Py drop)
```

arcjet-go consumes the output of step 2 (wizered, optionally opt'd).

### Regen

From a fresh arcjet-analyze checkout:

```sh
cd ../arcjet/arcjet-analyze/bindings_js_req
make build-wasm

cd ../../../arcjet-go
cp ../arcjet/arcjet-analyze/bindings_js_req/dist/arcjet_analyze_js_req.wasm \
   internal/local/jsreq/js_req.wasm
gravity --world js-req \
   --output internal/local/jsreq/bindings.go \
   internal/local/jsreq/js_req.wasm
sed -i '1,5 s/^package js_req$/package jsreq/' internal/local/jsreq/bindings.go
go test ./...
```

## Testing

In-package tests in `internal/local/jsreq/imports_test.go` exercise
the host callback defaults and the custom-detect callback wrapping
(arity-mismatch guard, custom label round-trip) — constructing the
unexported `defaultEmailOverrides{}` / `sensitiveInfoIdentifier{...}`
types directly.

End-to-end tests in `local_decision_test.go` and `guard_test.go` run
the factory + rules through wazero, asserting on the SDK-level
`Decision` / `GuardDecision` surface. Includes a belt-and-braces check
that raw sensitive-info text never appears on the Guard wire.
